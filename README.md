# El Classico Turf

A turf-ground booking SaaS for El Classico Turf's four Chennai-area branches
(Potheri, Chengalpattu, Selaiyur/Tambaram, Porur). Customers browse locations,
reserve an hourly slot for 5-a-side or 6-a-side turf, and pay online via
Razorpay (test mode). Admins get a control panel for live analytics, booking
management, manual slot blocking, and a customer directory.

**Live demo:** frontend on GitHub Pages — https://\<your-github-username>.github.io/elclassicoturfweb/
*(only fully functional once the backend below is deployed and `VITE_API_URL` is set — see [Deployment](#deployment))*

## Stack

- **Frontend**: React 19 + Vite 6 + TypeScript + Tailwind 4 (CSS-first `@theme`)
  + Framer Motion + React Router 7 (HashRouter).
- **Backend**: Express + TypeScript + Prisma ORM, running against Postgres
  (e.g. a free Supabase project). Auth via httpOnly JWT cookies + bcrypt.
  Payments via the Razorpay Node SDK (test mode) + Checkout.js on the client.

## Running locally

**Prerequisites:** Node.js 18+, a Postgres database (e.g. a free Supabase
project), and a Razorpay test-mode account.

1. Install dependencies for both the frontend and backend:
   ```
   npm install
   npm --prefix server install
   ```
2. Copy `server/.env.example` to `server/.env` and fill in `DATABASE_URL`,
   `JWT_SECRET`, `RAZORPAY_KEY_ID`, and `RAZORPAY_KEY_SECRET`.
3. Set up the database schema and seed data:
   ```
   npm --prefix server run prisma:migrate
   npm --prefix server run prisma:seed
   ```
   The seed script prints a default admin login (email/password) to the
   console — use it to sign in to `/admin`.
4. (Optional) Copy `.env.example` to `.env.local` at the project root if you
   need to point the frontend at an API hosted on a different origin.
5. Run both the frontend and backend together:
   ```
   npm run dev:all
   ```
   Or run them separately: `npm run dev` (frontend, port 3000) and
   `npm --prefix server run dev` (backend, port 4000).

## Payments

Bookings are created with `paymentStatus: "Pending"`. For UPI/Card payments,
the frontend opens Razorpay Checkout against a test-mode order created by the
backend; on success the backend verifies the payment signature and marks the
booking as `Paid`. "Pay at Ground" bookings stay `Pending` until an admin
marks them paid from the control panel.

## Deployment

The frontend and backend deploy separately: frontend → **GitHub Pages**,
backend → **Render** (or any Node host — Railway, Fly.io, etc. work the same way).

### 1. Backend → Render

1. In the [Render dashboard](https://dashboard.render.com), click **New →
   Blueprint**, connect this repo, and Render will pick up [`render.yaml`](render.yaml)
   automatically. It provisions a free web service rooted at `server/`.
2. Fill in the environment variables Render asks for (marked `sync: false` in
   the blueprint, so they aren't committed to git):
   | Variable | Value |
   |---|---|
   | `DATABASE_URL` | Supabase pooled connection string (port 6543, `?pgbouncer=true`) |
   | `DIRECT_URL` | Supabase direct connection string (port 5432) — used for migrations |
   | `RAZORPAY_KEY_ID` / `RAZORPAY_KEY_SECRET` | From your Razorpay test-mode dashboard |
   | `CORS_ORIGIN` | Your GitHub Pages URL, e.g. `https://<user>.github.io` |

   `JWT_SECRET` is auto-generated by Render; `PORT` is preset.
3. Deploy. Render runs `prisma migrate deploy` on every deploy, so schema
   changes roll out automatically. Run `npm --prefix server run prisma:seed`
   once (via the Render shell) to seed the 4 branches + admin account.
4. Note the resulting service URL, e.g. `https://elclassicoturfweb-api.onrender.com`.

### 2. Frontend → GitHub Pages

1. In the repo's **Settings → Pages**, set Source to **GitHub Actions**
   (the included [`deploy-pages.yml`](.github/workflows/deploy-pages.yml) workflow
   handles the rest).
2. In **Settings → Secrets and variables → Actions → Variables**, add a
   repository variable `VITE_API_URL` set to your Render backend URL + `/api`,
   e.g. `https://elclassicoturfweb-api.onrender.com/api`.
3. Push to `main` (or run the workflow manually from the **Actions** tab) —
   it builds the Vite app with the GitHub Pages base path and publishes `dist/`.

Free-tier Render services spin down when idle, so the first request after a
while may take ~30s to wake up.
